# 01. 업무 프로세스 입력 매뉴얼 (직접 입력)

이 문서는 업무 프로세스를 **표준 양식(`board-v1` JSON)** 으로 직접 작성하는
방법을 설명합니다. JSON을 처음 봐도 따라 할 수 있도록 단계별로 안내합니다.

> 💡 **빠르게 하고 싶으신가요?** 대부분의 경우 [`02-AI-입력-가이드`](02-AI-입력-가이드.md)
> 대로 업무를 말로 설명하면 AI가 이 문서의 결과물을 자동으로 만들어 줍니다.
> 이 문서는 그 결과물을 **이해하고 다듬기 위한** 참고서로도 유용합니다.

---

## 0. 준비 (최초 1회)

Node.js 20 이상이 설치된 PC에서:

```bash
git clone https://github.com/evelynn/Hansol-100.git
cd Hansol-100
npm install          # 의존성 설치 (최초 1회)
npm test             # 정상 설치 확인 (모든 테스트 통과여야 함)
```

PNG 이미지도 만들려면 `rsvg-convert`(librsvg) 또는 `cairosvg` 중 하나를
설치하세요(선택). 없으면 SVG만 생성됩니다.

## 1. 핵심 개념 — 프로세스를 격자로 생각하기

업무 프로세스를 **격자(표)** 로 봅니다.

```
              부서·담당 (레인, 좌 → 우)
              요청부서   부서장   구매팀   재무팀
   단계  S1 요청   [구매요청]
   (위    S2 승인             [검토·승인]
    →     S3 발주                      [발주]   [예산검토]
    아래) S4 검수                               [대금정산]
```

- **레인(lane)** = "누가" — 부서·담당·역할. 표의 **열**. 실제 업무가 넘어가는
  순서대로 왼쪽→오른쪽에 둡니다(요청자를 왼쪽, 최종 결재자를 오른쪽).
- **단계(stage)** = "언제" — 프로세스가 지나가는 국면. 표의 **행**. 위→아래 순서.
- **업무(node)** = "무엇을" — 한 칸(부서×단계) 안의 일 하나. **카드**로 그려짐.
- **연결(edge)** = 업무 사이의 흐름·정보 공유·반려.

이 네 가지를 JSON의 `lanes`, `stages`, `nodes`, `edges`에 채우면 됩니다.

## 2. 파일 구조 한눈에 보기

가장 작은 예시입니다. `templates/board.template.json`을 복사해 시작하세요.

```json
{
  "schema_version": 1,
  "profile": "hansol",
  "title": "보드 제목",
  "subtitle": "한 줄 설명",
  "lanes": ["요청부서", "부서장"],
  "stages": ["S1 요청", "S2 승인"],
  "nodes": [
    { "id": "N1", "lane": "요청부서", "stage": "S1 요청", "label": "요청서 작성", "emphasis": "lead" },
    { "id": "N2", "lane": "부서장",   "stage": "S2 승인", "label": "검토·승인", "emphasis": "key" }
  ],
  "edges": [
    { "id": "E1", "source": "N1", "target": "N2", "type": "sequence", "label": "상신" }
  ]
}
```

- `"profile": "hansol"` — 한솔 사내용 한글 스타일. (생략하면 영문 `default`)
- `title` / `subtitle` — 보드 상단 제목/부제.
- `lanes` / `stages` — 문자열 배열. **순서가 곧 화면 배치 순서**입니다.
- `nodes` / `edges` — 아래에서 자세히 설명합니다.

## 3. 필드 상세

### 3.1 노드(업무 카드) `nodes[]`

| 필드 | 필수 | 설명 |
|---|---|---|
| `id` | ✅ | 노드 고유 식별자. 겹치면 안 됨 (예: `N1`, `N2`). 연결(edge)에서 이 값을 참조. |
| `lane` | ✅ | 이 업무를 하는 부서·담당. **`lanes` 배열에 있는 값과 정확히 일치**해야 함. |
| `stage` | ✅ | 이 업무가 속한 단계. **`stages` 배열에 있는 값과 정확히 일치**해야 함. |
| `label` | ✅ | 카드에 크게 표시될 업무명. 짧은 동사구 권장 (예: "부서장 검토·승인"). |
| `emphasis` | 선택 | 업무의 성격/중요도. `lead`/`key`/`normal`/`bottleneck`/`loop` 중 하나. 생략 시 `normal`. |
| `note` | 선택 | 카드 본문에 작게 표시될 부연 설명(한 문장). |
| `refs` | 선택 | 근거 목록. `[{ "source": "구매관리규정 제5조", "note": "..." }]` 형태. `근거` 라벨 아래 표시. |

### 3.2 emphasis — 업무 성격 (색·배지 결정)

`hansol` 프로파일 기준 배지와 의미:

| `emphasis` | 배지 | 언제 쓰나 |
|---|---|---|
| `lead` | **주관** | 그 단계를 시작·주도하는 업무 (첫 착수, 기안). |
| `key` | **핵심** | 프로세스에서 가장 결정적인 업무 (승인/결정 등). |
| `normal` | **진행** | 일상적인 통상 업무. 기본값. 대부분 여기 해당. |
| `bottleneck` | **지연** | 자주 막히거나 시간이 오래 걸리는 업무. |
| `loop` | **반려** | 뒤 단계에서 되돌아오는 업무 (반려·재작업·재상신). |

> ⚠️ **과하게 표시하지 마세요.** 모든 카드를 `key`/`lead`로 두면 강조가 의미를
> 잃습니다. 한 단계에 `lead`/`key`는 1~2개면 충분하고, `bottleneck`/`loop`는
> 실제로 느리거나 되돌아오는 업무에만 씁니다.

### 3.3 연결(화살표) `edges[]`

| 필드 | 필수 | 설명 |
|---|---|---|
| `id` | ✅ | 연결 고유 식별자 (예: `E1`, `L1`). |
| `source` | ✅ | 시작 노드의 `id`. 실제 존재하는 노드여야 함. |
| `target` | ✅ | 도착 노드의 `id`. 실제 존재하는 노드여야 함. |
| `type` | 선택 | `sequence`/`message`/`loop` 중 하나. 생략 시 `sequence`. |
| `label` | 선택 | 화살표 위 라벨 (예: "상신", "승인", "반려"). |

**연결 종류(`type`):**

- `sequence` — 정상적인 앞으로의 흐름. 기본값. (실선 화살표)
- `message` — 부서 간 **정보 공유·통보** (단계 전환 없이 알림 성격). (초록 점선)
- `loop` — **반려·재작업·재상신** 되돌아오는 경로. 도착 노드의 `emphasis`도
  `loop`(반려)로 두면 카드와 선 양쪽에서 회귀가 잘 보입니다. (파란 점선)

## 4. 따라 만들기 — 구매 요청 프로세스

다음 업무를 예로 보드를 완성해 봅니다.

> 요청부서가 구매 요청서를 올리면 부서장이 검토·승인한다. 반려되면 요청부서가
> 보완해 다시 올린다. 승인되면 구매팀이 업체를 선정하고, 재무팀이 예산을 검토한
> 뒤 구매팀이 발주서를 발행한다. 물품이 오면 요청부서가 검수하고 재무팀이 대금을
> 지급·정산한다.

### 4단계 절차

**① 레인(부서) 뽑기** — 등장하는 부서를 업무 순서대로:
`요청부서`, `부서장`, `구매팀`, `재무팀`

**② 단계 뽑기** — 프로세스가 지나가는 국면을 순서대로. 번호 접두어(S1, S2…)를
붙이면 순번이 잘 읽힙니다:
`S1 요청`, `S2 승인`, `S3 발주`, `S4 검수·정산`

**③ 업무(노드) 배치** — 각 부서×단계 칸에 업무 하나씩. 성격에 맞는 `emphasis`:
- `요청부서 / S1`: 구매 요청서 작성·제출 → `lead`(주관)
- `부서장 / S2`: 부서장 검토·승인 → `key`(핵심)
- `요청부서 / S2`: 반려 사유 보완·재상신 → `loop`(반려)
- `구매팀 / S3`: 업체 선정·견적 비교 → `normal`
- `재무팀 / S3`: 예산·지출 검토 → `bottleneck`(지연)
- `구매팀 / S3`: 발주서 발행 → `normal`
- `요청부서 / S4`: 납품 검수 → `normal`
- `재무팀 / S4`: 대금 지급·정산 → `normal`

**④ 연결(화살표) 잇기** — 흐름·반려·통보:
- 요청 → 승인: `sequence` "상신"
- 승인 → 보완: `loop` "반려"
- 보완 → 승인: `sequence` "재상신"
- 승인 → 업체선정: `sequence` "승인"
- 업체선정 → 예산검토 → 발주: `sequence`
- 발주 → 검수: 정보 통보는 `message`, 실제 흐름은 `sequence`
- 검수 → 정산: `sequence` "검수 완료"

완성된 전체 파일은 이 저장소의
[`fixtures/hansol-sample.json`](../../fixtures/hansol-sample.json) 과 동일합니다.
그대로 열어 참고하거나 복사해 수정하세요.

## 5. 렌더링 · 점검 · 반복

파일을 `my-process.json`으로 저장했다면:

```bash
# 1) 그림 그리기 (SVG, 가능하면 PNG도)
node scripts/board.mjs render my-process.json --out my-process.svg --png

# 2) 품질 점검 — nodePiercings 는 반드시 0 이어야 함
node scripts/board.mjs audit my-process.json

# 3) 기준 통과 여부 게이트 (자동화/제출 전 확인)
node scripts/board.mjs validate my-process.json --strict

# 4) (선택) 단계 리빌 애니메이션
node scripts/board.mjs motion my-process.json --out my-process.motion.svg
```

`audit` 출력 예시와 해석:

```
score: 7.5
  nodePiercings: 0     ← 0 이어야 정상 (선이 카드 뒤로 가려짐 = 0)
  crossings: 2         ← 선 교차. 복잡한 절차면 약간은 정상
  bendsPerEdgeMax: 2   ← 한 선의 최대 꺾임 수
  routeStretchMax: 0.92
```

`nodePiercings`가 0보다 크면 4장으로 돌아가 같은 칸에 너무 많은 노드가 몰렸는지,
한 선이 너무 여러 단계를 건너뛰는지 확인하세요. 자세한 지표 설명:
[`references/composition-quality.md`](../../references/composition-quality.md).

## 6. 자주 나는 오류와 해결

| 증상 / 메시지 | 원인 | 해결 |
|---|---|---|
| `lane "..." not in lanes` | 노드의 `lane` 값이 `lanes` 배열에 없음 (오타·띄어쓰기) | 문자열을 **정확히** 일치시키기 |
| `stage "..." not in stages` | 노드의 `stage` 값이 `stages`에 없음 | 위와 동일 |
| `edge source/target ... unknown` | 연결이 없는 노드 `id`를 참조 | `source`/`target`을 실제 `id`로 |
| `duplicate node id` | `id`가 중복됨 | 모든 `id`를 고유하게 |
| `Unexpected token ... in JSON` | 쉼표 빠짐·따옴표 오류 등 JSON 문법 오류 | 마지막 항목 뒤 쉼표 제거, 큰따옴표 사용 |
| 배지가 영문(Lead/Key)으로 나옴 | `profile`이 `hansol`이 아님 | `"profile": "hansol"` 추가 또는 `--profile hansol` |

## 7. 제출 전 체크리스트

- [ ] `"profile": "hansol"` 설정됨 (사내 표준 스타일)
- [ ] 모든 노드 `id`가 고유함
- [ ] 모든 `lane`/`stage`가 배열의 값과 정확히 일치함
- [ ] 모든 `edge`의 `source`/`target`이 실제 노드를 가리킴
- [ ] `emphasis` 강조가 과하지 않음 (단계당 핵심/주관 1~2개)
- [ ] 반려·재작업 경로가 `loop`로 표시됨
- [ ] `validate --strict` 통과, `audit`의 `nodePiercings`가 0
- [ ] 근거가 필요한 업무에 `refs`(규정·전결) 기재됨
